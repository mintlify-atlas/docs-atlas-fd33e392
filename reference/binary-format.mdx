Automerge uses a specialized binary format for efficient storage and transmission of CRDT data. This format provides compact representation of documents and changes.

## Overview

The Automerge binary format is a columnar compression format designed to:

- **Minimize storage size** - Typical documents are 10-100x smaller than JSON
- **Enable fast loading** - Columnar format allows efficient parsing and decompression
- **Support incremental sync** - Changes can be efficiently transmitted over networks
- **Maintain history** - Complete operation history is preserved

## Key Concepts

### Columnar Compression

Automerge 3.0 introduced columnar compression that operates both at runtime and on disk:

- **Memory efficiency** - 100x lower steady-state memory usage compared to previous versions
- **Predictable from disk size** - Runtime memory usage correlates with saved document size
- **Runtime optimization** - The same compression used for storage is used in memory

```javascript
import * as automerge from "@automerge/automerge"

const doc = automerge.from({ tasks: [] })

// Save in compressed binary format
const binary = automerge.save(doc)
console.log(`Document size: ${binary.byteLength} bytes`)

// Load from binary
const loaded = automerge.load(binary)
```

### Document Structure

The binary format encodes:

1. **Operations** - All CRDT operations that created the document
2. **Actor IDs** - Unique identifiers for each peer that made changes
3. **Causality information** - Dependency relationships between operations
4. **Object structure** - Maps, lists, text, and other data types
5. **Metadata** - Timestamps, commit messages, and other annotations

### Changes vs. Complete Documents

Automerge distinguishes between:

**Complete document format:**
```javascript
const binary = automerge.save(doc)
// Contains all operations from document creation
```

**Individual changes:**
```javascript
const changes = automerge.getChanges(oldDoc, newDoc)
const binary = automerge.encodeChange(changes[0])
// Contains only the operations in this change
```

**Change bundles:**
```javascript
const bundle = automerge.bundle(changes)
// Efficiently encodes multiple changes together
```

## Storage and Compression

### Saving Documents

Use `save()` to serialize a document:

```javascript
import * as automerge from "@automerge/automerge"

const doc = automerge.from({
  title: "My Document",
  content: "Hello, world!"
})

const binary = automerge.save(doc)

// Save to file, database, or send over network
await saveToDatabase(binary)
```

### Loading Documents

Load binary data back into a document:

```javascript
const binary = await loadFromDatabase()
const doc = automerge.load(binary)

console.log(doc.title) // "My Document"
```

### Incremental Loading

For large documents, load incrementally:

```javascript
let doc = automerge.init()

// Load changes in batches
for (const changeBatch of changeIterator) {
  doc = automerge.applyChanges(doc, changeBatch)
}
```

## Performance Characteristics

### Memory Usage (Automerge 3.0+)

<Note>
Automerge 3.0 achieved approximately 10x reduction in memory usage through columnar compression.
</Note>

Memory usage is now:
- **Predictable** - Correlates directly with document size on disk
- **Efficient** - Roughly 2-3x the saved binary size in RAM
- **Scalable** - Can handle much larger documents than before

### Compression Ratios

Typical compression ratios compared to JSON:

| Data Type | Compression Ratio |
|-----------|------------------|
| Text documents | 10-50x |
| Structured data | 5-20x |
| Rich text | 15-40x |
| Lists with history | 20-100x |

### Load Times

Loading performance depends on:
- Document size (operations count)
- Number of objects
- History depth
- Hardware capabilities

Typical load times:
- Small docs (less than 1MB): under 10ms
- Medium docs (1-10MB): 10-100ms
- Large docs (10-100MB): 100ms-1s

## Working with Binary Data

### Encoding Changes

```javascript
import * as automerge from "@automerge/automerge"

let doc1 = automerge.init()
doc1 = automerge.change(doc1, d => {
  d.message = "Hello"
})

let doc2 = automerge.clone(doc1)
doc2 = automerge.change(doc2, d => {
  d.message = "World"
})

// Get changes made to doc2
const changes = automerge.getChanges(doc1, doc2)

// Encode for transmission
const encoded = changes.map(c => automerge.encodeChange(c))
```

### Decoding Changes

```javascript
// Receive encoded changes
const receivedBinary = await receiveFromNetwork()

// Decode and apply
const change = automerge.decodeChange(receivedBinary)
doc1 = automerge.applyChanges(doc1, [change])
```

### Bundling Changes

Bundle multiple changes for efficient storage:

```javascript
const changes = automerge.getAllChanges(doc)
const bundle = automerge.bundle(changes)

// Bundle is more compact than individual changes
console.log(`Individual: ${totalSize(changes)} bytes`)
console.log(`Bundled: ${bundle.byteLength} bytes`)
```

## Network Transmission

For network sync, use the sync protocol (recommended) or send raw changes:

### Using Sync Protocol (Recommended)

```javascript
import * as automerge from "@automerge/automerge"

let doc = automerge.init()
let syncState = automerge.initSyncState()

// Generate sync message
const [newSyncState, message] = automerge.generateSyncMessage(doc, syncState)
syncState = newSyncState

// Send message over network
await sendToNetwork(message)

// Receive and process sync message
const receivedMessage = await receiveFromNetwork()
const [newDoc, newSyncState2, patch] = automerge.receiveSyncMessage(
  doc, 
  syncState, 
  receivedMessage
)
```

### Sending Individual Changes

```javascript
// Get new changes
const changes = automerge.getChanges(oldDoc, newDoc)

// Encode and send
for (const change of changes) {
  const binary = automerge.encodeChange(change)
  await sendToNetwork(binary)
}
```

## Migration and Compatibility

### Version Compatibility

The binary format is versioned:
- **Automerge 2.x** - Compatible with 1.x format
- **Automerge 3.x** - New columnar format, can read 2.x documents

<Warning>
Automerge 3.x introduces a new binary format. While it can read documents from earlier versions, documents saved with 3.x may not be readable by 2.x clients.
</Warning>

### Converting Documents

To convert from older formats:

```javascript
// Load document from any version
const doc = automerge.load(oldBinary)

// Save in current format
const newBinary = automerge.save(doc)
```

## Advanced Topics

### Verification Mode

When loading documents, you can verify integrity:

```javascript
// Load with verification (slower but safer)
const doc = automerge.load(binary, {
  verificationMode: "full"
})
```

### Custom Storage Backends

Implement custom storage by working with binary data:

```javascript
class DocumentStore {
  async save(docId, doc) {
    const binary = automerge.save(doc)
    await this.database.put(docId, binary)
  }
  
  async load(docId) {
    const binary = await this.database.get(docId)
    return automerge.load(binary)
  }
}
```
